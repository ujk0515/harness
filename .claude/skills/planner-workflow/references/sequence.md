# Planner Sequence

## 1. 입력 정리
1. `request-workboard.md`에서 현재 item을 찾는다.
2. `project-config.md`에서 플랫폼과 스택을 확인한다.
3. `A-benchmark.md`가 있으면 참고한다.
4. revise면 직전 계획 md와 리뷰 문서를 함께 읽는다.

## 2. 파일 만들기
1. 새 계획 md 파일명을 정한다.
2. 기본은 `plan_{item_id}_{short_title}.md`
3. 재계획이면 `_r2`, `_r3` 같은 suffix를 붙인다.

## 3. 본문 작성
1. 사용자 원문
2. 목표
3. 기능 명세
4. 화면/플로우 요약
5. ASCII 화면
6. 요소 디스크립션
7. 동작·상태
8. 실행·검증 메모
9. 완료 기준
10. 비범위
11. 메모

## 3-1. 실행·검증 메모 작성 기준
- 실행 가능한 프로젝트일 때만 채운다.
- 최소 포함:
  - 프론트 명령
  - 서버 명령
  - 기본 URL
  - API URL
  - smoke 명령
  - full 명령 (있을 때)
  - diag 명령 (있을 때)
  - item 범위 target spec 또는 grep
  - 스크린샷/리포트 경로
  - 권장 검증 명령
- 포트나 명령을 모르면 추측으로 늘리지 말고 `미정`으로 적고 후속 확인 대상으로 남긴다.
- Playwright를 쓰는 repo라면 `baseURL`, `webServer`, `projects` 분리 여부를 짧게 적는다.

## 4. ASCII 작성 기준
- 기본 화면과 상태 화면을 분리한다.
- CTA, 입력, 목록, 모달 같은 핵심 블록을 드러낸다.
- 너무 자세한 픽셀 설명 대신 구조와 우선순위를 전달한다.

## 5. revise 기준
- 직전 파일을 덮어쓰지 않는다.
- 새 파일에서 바뀐 점을 먼저 적는다.
- 리뷰에서 나온 지적이 있으면 해당 섹션에 바로 반영한다.
